Lab 8 pygame project | Focus: random motion, bounce logic, and game loop structure.

Runtime Walkthrough

The program initializes pygame, creates square objects, then repeats an event-update-render loop at 60 FPS. Movement comes from velocity plus random drift, while wall collision inverts velocity to keep squares on screen.

Good Patterns
✅Named constants for game config
Screen size, count, and colors are constants at the top, making tuning safer and easier to read.
Basics:One edit changes behavior globally.
Engineering Insight:Fewer magic numbers reduces accidental mismatch bugs.
Architecture Insight:Separates policy values from runtime logic flow.
Python constants style
✅Square class owns motion and drawing
Each square tracks its own state and exposes update and draw, reducing logic spread.
Basics:Behavior stays close to the data it changes.
Engineering Insight:Encapsulation makes adding features like trails easier.
Architecture Insight:Entity style design scales to many object types.
Encapsulation reference
✅Clear event update render loop
The frame loop follows a standard game architecture: input first, simulation second, render last.
Basics:The order is predictable each frame.
Engineering Insight:Predictable order simplifies debugging state issues.
Architecture Insight:This is a lightweight main loop engine pattern.

Potential Issues
⚠️Frame rate tied movement
Movement uses fixed per frame velocity, so speed changes if FPS changes on slower hardware.
Basics:Different computers can produce different motion speeds.
Engineering Insight:Use delta time to make physics frame independent.
Architecture Insight:Time step handling is key in simulation architecture.
Hint: multiply velocity by elapsed seconds from clock.tick().
pygame timing docs
⚠️No input validation layer
Constants are assumed valid; negative sizes or invalid colors could break rendering if edited carelessly.
Basics:Bad config values can crash or draw incorrectly.
Engineering Insight:Guard checks fail fast with helpful error messages.
Architecture Insight:Validation boundary protects engine internals.
Hint: add a small startup function that validates width, height, and square size.

Performance Wins
Bounded object count
The loop complexity is linear in NUM_SQUARES, and default count is small for real-time rendering.
Single frame flip strategy
Only one display flip per frame keeps rendering synchronized and avoids extra buffer swaps.
Performance Risks
Random generation in hot loop
Two random.uniform calls per square per frame can become expensive at high entity counts.
Repeated rect creation
A new pygame.Rect is built every draw call; this is fine now but scales poorly with many sprites.
4 / 6 Verdict: good for a small demo, but optimize update/draw hot path before scaling to hundreds of objects.

Beginner Debugging Tips

1
Print key state
Temporarily print x, y, vx, and vy for one square every 30 frames to confirm bounce logic.
2
Inspect events
Log event.type values to verify quit events are actually reaching your loop.
3
Control randomness
Add random.seed(42) before object creation when you need repeatable behavior during debugging.
4
Track frame timing
Capture dt_ms = clock.tick(60) and print it occasionally to detect FPS drops.
